# Prompt para Cursor — Dashboard simplificado v1.8.0 (solo UI)

Iteración de UI del dashboard. Pivote de "workspace multi-pieza" a **una sola vista enfocada: lista priorizada de leads enfriándose**. Bump `v1.7.0 → v1.8.0`.

**Alcance estricto: solo UI**. El frontend consume un mock estático que simula la respuesta del backend. Toda la lógica de criterio, filtrado y ordenamiento queda en backend (documentada al final como especificación). No agregar fórmulas, umbrales ni cómputos de negocio al cliente.

Asume que v1.7.0 (rediseño visual del detalle a 3 columnas) ya está aplicado. **Este prompt no toca el detalle del lead**.

## Decisiones cerradas (UI)

1. Eliminar tabs "Actividades" y "Leads". Vista única.
2. Eliminar bloque "Tu cobertura" completo.
3. Eliminar botón "Filtros" y todo el state asociado.
4. Eliminar pill "Pregunta cualquier cosa..." y todo el bloque del asistente del dashboard.
5. Header del shell mínimo: solo "Esc para cerrar" + X.
6. Tabla simplificada a 4 columnas:
   - **Familia** — contacto + ubicación arriba, relación + persona cuidada + edad abajo
   - **Tipo** — tipo de servicio del lead
   - **Por qué** — razón concreta del enfriamiento
   - **Sin contacto** — días grande + fecha del último mensaje pequeño debajo
7. Tope visual: **5 leads** (`MAX_VISIBLE = 5`, constante de UI).
8. Botón "Refrescar" debajo de la lista. Click → vuelve a llamar al servicio. Los completados en sesión quedan filtrados client-side.
9. Estado vacío / al día: `"Estás al día. No hay leads enfriándose."`
10. Degradación de color por urgencia (presentational, basado en `daysWithoutTouch`):
    - 0-3: normal
    - 4-7: amarillo muy sutil
    - 8-14: naranja sutil
    - 15-20: rojo sutil
    - 21+: rojo + ícono `alert-triangle` antes del número
11. Acciones por fila:
    - Click en celda (excepto botón) → abre el detalle del lead
    - Hover muestra botón "Marcar hecho" → click ejecuta directo + toast inferior `"Marcado como hecho · Deshacer"` durante 4s

### 11. Handlers nuevos

```javascript
case 'mark-done': {
    state.completedTodayIds.add(id);
    
    state.undoToast = {
        leadId: id,
        expiresAt: Date.now() + 4000
    };
    
    if (state.undoTimeoutId) clearTimeout(state.undoTimeoutId);
    state.undoTimeoutId = setTimeout(function () {
        state.undoToast = null;
        rerenderContent();
    }, 4000);
    
    rerenderContent();
    return;
}

case 'undo-mark-done': {
    if (state.undoToast) {
        state.completedTodayIds.delete(state.undoToast.leadId);
        state.undoToast = null;
        if (state.undoTimeoutId) clearTimeout(state.undoTimeoutId);
    }
    rerenderContent();
    return;
}

case 'refresh-cooling': {
    // Re-render llama a getCoolingLeads() nuevamente.
    // Los completados quedan filtrados naturalmente.
    rerenderContent();
    return;
}
```

## TODOs para backend (especificación, NO implementar en frontend)

Esta sección documenta las decisiones de negocio que el endpoint debe cumplir cuando se cablee. **No agregar esta lógica al frontend en esta iteración.**

### Endpoint

`GET /api/me/leads/cooling`

### Cohorte (criterio operativo de "enfriándose")

A confirmar con Pablo Rodrigo + Adriana Barro:
- Lead dentro de ventana de **15 días** desde la primera llamada (`entryDate`).
- `daysWithoutTouch >= 4` (umbral configurable).
- **NO tiene `nextScheduledFollowUp` con fecha futura** (si la AF ya agendó próximo contacto, el lead no está enfriándose por inacción, está "esperando por diseño" — ej: lead con interna que arranca dentro de un mes).
- NO histórico, NO convertido.
- NO filtrar por `temperature` del lead.

### Orden

- Primario: `daysWithoutTouch` descendente.
- Secundario: proximidad a salida de ventana de 15 días.

### Forma del response

```json
[
    {
        "leadId": "L121399",
        "contact": "Marta Ortiz",
        "location": "Madrid",
        "relation": "Madre",
        "caredPersonName": "Concepción",
        "age": 86,
        "serviceType": "Interno",
        "daysWithoutTouch": 22,
        "lastMessageDate": "27-abr",
        "reason": "Pidió pensarlo en familia, sin respuesta"
    }
]
```

Notas:
- `reason` puede ser derivado de la actividad de Odoo o ser una columna agregada manualmente por la AF.
- `lastMessageDate` viene formateado para visualización ("27-abr"). Si el backend prefiere ISO, el frontend formatea — pero agrega lógica que se prefiere evitar.
- `daysWithoutTouch` viene calculado del backend.

### Endpoint de "marcar hecho" (futuro)

`POST /api/leads/:leadId/mark-done` (sugerido).

Cuando esté disponible, el handler `mark-done` del frontend hace la llamada **después del undo de 4s** (optimistic UI con confirmación post-undo-window). Si falla, se restaura la fila y se muestra error. Por ahora en MVP solo state local de sesión.

### Endpoint de "refrescar"

No requiere endpoint adicional. `Refrescar` simplemente vuelve a llamar a `GET /api/me/leads/cooling`.
