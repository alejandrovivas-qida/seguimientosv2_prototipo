# Widget de Seguimientos — Fase A: Lead Detail desde Odoo

> **Audiencia**: agente de código.
> **Objetivo**: reemplazar los mocks de datos del lead por llamadas reales a Odoo desde el widget (Tampermonkey, same-origin con `erp.qida.es`).
> **NO toca UI**: el HTML/CSS y la estructura de paneles del widget ya están. Solo se cablea la lógica de fetch + se popula el modelo de datos del widget.
> **Backend**: NO requerido en esta fase.

## 1. Contexto mínimo

Qida coordina cuidado de personas dependientes. Las AFs (Asesoras Familiares) gestionan leads (familias). El widget vive embedded en Odoo y muestra un modal de Lead Detail con datos de la familia, persona cuidada, notas, actividades, etc.

Hoy esos datos están **mockeados** en el widget. Fase A los reemplaza por datos reales pegándole directo a Odoo.

- Pilotos (3 champions): widget inyectado vía **Tampermonkey** sobre `erp.qida.es`.
- Same-origin con Odoo → cookie `session_id` viaja automática en cada fetch.
- Sin CORS, sin CSRF, sin backend en esta fase.

## 4. Helpers únicos

```js
// Módulo del widget: contexto base de la sesión.
// Se inicializa una vez (sección 5) y se mergea en cada llamada.
let _baseContext = {};

async function odooCall(model, method, args = [], kwargs = {}) {
  const url = `/web/dataset/call_kw/${model}/${method}`;

  // Mergeo de contexto: el base (lang, tz, allowed_company_ids) + el que pase el caller
  const mergedContext = { ..._baseContext, ...(kwargs.context || {}) };
  const finalKwargs = { ...kwargs, context: mergedContext };

  let res;
  try {
    res = await fetch(url, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        jsonrpc: "2.0",
        method: "call",
        params: { model, method, args, kwargs: finalKwargs }
      })
    });
  } catch (networkErr) {
    throw new Error(`Odoo ${model}/${method}: network error — ${networkErr.message}`);
  }

  // Defensivo: si la sesión expiró Odoo puede devolver HTML del login (no JSON)
  if (!res.ok) {
    if (res.status === 401 || res.status === 403) {
      throw new Error(`Odoo session expired (HTTP ${res.status}). Redirigir a /web/login.`);
    }
    throw new Error(`Odoo ${model}/${method}: HTTP ${res.status}`);
  }

  const contentType = res.headers.get('content-type') || '';
  if (!contentType.includes('application/json')) {
    // Probablemente HTML del login
    throw new Error(`Odoo ${model}/${method}: respuesta no-JSON (sesión probablemente expirada).`);
  }

  const data = await res.json();
  if (data.error) {
    throw new Error(`Odoo ${model}/${method}: ${data.error.data?.message || data.error.message}`);
  }
  return data.result;
}

async function odooSession() {
  const res = await fetch('/web/session/get_session_info', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ jsonrpc: "2.0", method: "call", params: {} })
  });

  if (!res.ok) {
    throw new Error(`Odoo session_info: HTTP ${res.status}`);
  }
  const contentType = res.headers.get('content-type') || '';
  if (!contentType.includes('application/json')) {
    throw new Error(`Odoo session_info: respuesta no-JSON (sesión probablemente expirada).`);
  }
  const data = await res.json();
  return data.result;
}
```

Toda llamada a Odoo pasa por estos dos helpers. No hacer `fetch` sueltos.

## 8. Notas internas (`mail.message`)

```js
const notes = await odooCall('mail.message', 'search_read', [], {
  domain: [
    ["model", "=", "crm.lead"],
    ["res_id", "=", LEAD_ID],
    ["subtype_id.internal", "=", true]
  ],
  fields: ["id", "author_id", "date", "body", "message_type", "subject"],
  order: "date desc",
  limit: 50
});
```

⚠️ `body` viene como **HTML** pre-renderizado por Odoo (tablas, divs, mentions). **NUNCA hacer `innerHTML` directo**. Sanitizar siempre. Helper defensivo:

```js
function sanitizeOdooHtml(rawHtml) {
  if (!rawHtml) return '';

  // 1) Preferí DOMPurify si está disponible
  if (typeof window.DOMPurify !== 'undefined') {
    return window.DOMPurify.sanitize(rawHtml, {
      ALLOWED_TAGS: ['p', 'br', 'b', 'strong', 'i', 'em', 'u', 'a', 'ul', 'ol', 'li', 'span', 'div'],
      ALLOWED_ATTR: ['href', 'target', 'rel'],
      ALLOW_DATA_ATTR: false
    });
  }

  // 2) Fallback defensivo sin DOMPurify: pasar por un DOM detached y eliminar script/style/event handlers
  const tpl = document.createElement('template');
  tpl.innerHTML = rawHtml;

  // Eliminar tags peligrosos
  tpl.content.querySelectorAll('script, style, iframe, object, embed, link, meta').forEach(el => el.remove());

  // Eliminar atributos on* y href javascript:
  tpl.content.querySelectorAll('*').forEach(el => {
    [...el.attributes].forEach(attr => {
      const name = attr.name.toLowerCase();
      const value = attr.value.toLowerCase();
      if (name.startsWith('on')) el.removeAttribute(attr.name);
      if ((name === 'href' || name === 'src') && value.startsWith('javascript:')) {
        el.removeAttribute(attr.name);
      }
    });
  });

  return tpl.innerHTML;
}

// Uso:
noteEl.innerHTML = sanitizeOdooHtml(note.body);
```

Si DOMPurify no está cargado en el ambiente Tampermonkey, considerar agregarlo vía `@require` en el header del userscript.

## 14. Gotchas técnicos

1. **Teléfonos con espacios**: Odoo devuelve `"+34 911 22 33 44"`. Para mostrar en UI está OK. Para Fase B (Timelines) normalizar con `tel.replace(/\s+/g, '').replace(/-/g, '')`.
2. **HTML en notas (`mail.message.body`)**: pre-renderizado por Odoo. SIEMPRE pasar por `sanitizeOdooHtml()` antes de `innerHTML`. Nunca raw.
3. **Sesión expirada → respuesta HTML, no JSON**. El helper `odooCall` lo detecta vía `content-type` y `response.ok`. Capturar el error y redirigir a `/web/login`.
4. **`fields: []`**: rompe por computed fields rotos de Qida. Siempre lista explícita.
5. **Multi-company**: el `_baseContext` se inicializa con `session.user_context` desde el init. Si en algún caso faltan leads esperados, agregar `allowed_company_ids` específico en `kwargs.context` de esa llamada.
6. **Endpoints `/agency/*`**: custom Qida fuera de servicio. **No usar**. Solo `/web/dataset/call_kw/...` y `/web/session/get_session_info`.
7. **Tampermonkey + CSP**: si Odoo tiene Content Security Policy estricto, podría bloquear scripts. Probar primero. Si pasa, usar `@grant` apropiados o cargar el widget desde un CDN propio.
8. **Tuples Odoo**: campos relacionales (`partner_id`, `user_id`, `stage_id`, etc.) vienen como `[id, "nombre"]`. Para mostrar texto usar `[1]`, para query usar `[0]`. Si el campo es vacío puede venir como `false` (no `null`).
9. **`chronich_illness`**: typo en el modelo de Odoo de Qida. Respetar el typo en el código, sino la llamada falla.
10. **DOMPurify**: si no está disponible, agregarlo via `@require https://cdn.jsdelivr.net/npm/dompurify@3.x/dist/purify.min.js` en el header del userscript de Tampermonkey. El fallback defensivo de `sanitizeOdooHtml` cubre lo básico, pero DOMPurify es la opción correcta.
